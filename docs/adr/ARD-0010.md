# ARD-0010 — Database Enum Strategy

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Database Schema
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA, RTM, user stories, permissions, UX, tech stack, roadmap, ARDs

## Context

GarageOS enums must stay lowercase, stable, API-safe, and schema-aligned. The schema permits PostgreSQL enums or `text` plus `CHECK` constraints.

## Decision

Use **`text` columns with named `CHECK` constraints** for initial business status and workflow enums.

Rules:

- Values must exactly match documented API/schema enum values.
- Constraints must be migration-controlled.
- Shared TypeScript enum/const definitions must use the same canonical source as API DTOs.

## Consequences

- Easier enum changes during initial build.
- Database still rejects invalid values.
- Requires drift control across schema, API, and frontend types.

## Validation

- Schema tests reject invalid enum values.
- API tests accept/return only documented enum values.
- Drift checks compare DTO enums with migration/check-constraint source.

## Risks

- TypeScript/API constants may drift from DB constraints.
- Constraint changes require careful migration sequencing.
- Inconsistent names can break API/schema alignment.

## Follow-ups

- Create canonical TypeScript/API enum source.
- Add enum-vs-check-constraint drift tests.
- Document enum-change migration procedure.
