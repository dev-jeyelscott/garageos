# ARD-0005 — Backend Framework and Modular Monolith

**Status:** Accepted
**Date:** 2026-06-24
**Owner:** Engineering
**Type:** Backend Architecture
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA plan, RTM, user stories, permission matrix, UX map, tech stack, roadmap, architecture records

## Context

GarageOS has many workflows that require atomic multi-table transactions, including inventory consumption, invoice issuance, payments/receipts, transfers, purchases, refunds, exports, deletion jobs, and supplier returns.

## Decision

Use **NestJS with TypeScript** as a modular monolith.

Organize backend modules as:

```text
module/
  api/
  application/
  domain/
  persistence/
  policies/
  events/
  tests/
```

Modules must communicate through application services and domain events/outbox records, not direct cross-module table mutation.

## Consequences

- Preserves transaction safety and implementation simplicity.
- Supports clear module boundaries and future extraction if justified.
- Requires discipline to prevent tight coupling.

## Validation

- Review or linting enforces module dependency rules.
- Critical commands use explicit transaction wrappers.
- Integration tests verify rollback behavior across multi-table workflows.

## Risks

- Boundaries may erode through cross-module table mutation.
- Large modules can become hard to maintain without dependency rules.
- Future extraction is harder if domain boundaries stay unclear.

## Follow-ups

- Define backend module dependency rules.
- Add shared transaction wrapper before high-risk workflows.
- Document event/outbox usage between modules.
