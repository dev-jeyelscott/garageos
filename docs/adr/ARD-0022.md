# ARD-0022 — OpenAPI Generation and Contract Drift Control

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** API Documentation / CI
**Required Before:** Milestone 2
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

`api-contracts.md` is canonical, but OpenAPI YAML is not yet generated. API implementation must stay contract-aligned.

## Decision

Generate OpenAPI from NestJS route metadata and DTO/schema annotations. Validate generated output against `api-contracts.md` through CI contract tests.

Rules:

- Check generated OpenAPI into the repo or publish it as a CI artifact.
- Require PR review for public API changes against API contracts and RTM trace.
- Contract tests must cover envelopes, auth headers, idempotency headers, pagination, enum values, status codes, and error semantics.

## Consequences

This reduces manual OpenAPI drift, but requires disciplined DTO annotations. The human-readable API contract remains the primary design source until generated OpenAPI is stable and reviewed.

## Validation

- CI fails when OpenAPI changes lack an approved artifact/update.
- API contract tests cover representative endpoints across modules.
- Frontend typed API client is generated or validated from stable OpenAPI.

## Risks

- Generated OpenAPI may miss runtime guards or business errors.
- DTO annotations may drift from `api-contracts.md`.
- Client generation may break while API surfaces are unstable.

## Follow-ups

- Add OpenAPI generation to CI after API skeleton.
- Define artifact path under `docs/api`.
- Validate generated client once endpoints stabilize.
