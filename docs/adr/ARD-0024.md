# ARD-0024 — Observability, Backup, and Disaster Recovery

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Operations / Reliability
**Required Before:** Milestone 13
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS requires structured logs, metrics, error monitoring, correlation IDs, background-job visibility, encrypted backups, quarterly restore tests, **RPO 24h**, and **RTO 4h**.

## Decision

Implement observability and recovery as production requirements:

- Structured JSON logs with request ID, correlation ID, safe tenant context, actor type, endpoint/action, outcome, and error code.
- OpenTelemetry-compatible traces/metrics where practical.
- Error monitoring for API, PWA, workers, providers, auth failures, authorization denials, exports, reminders, and inventory failures.
- Background-job dashboards/status views.
- Daily encrypted PostgreSQL backups.
- Object-storage lifecycle, backup, and retention strategy aligned to tenant deletion/retention rules.
- Quarterly restore tests.
- RPO target: **24 hours**.
- RTO target: **4 hours**.

Sensitive data must not be logged, over-exported, or exposed in error payloads.

## Consequences

This improves support and incident response, but adds launch-critical operational setup and requires strict redaction/access controls.

## Validation

- QA/DevOps verify correlation IDs and safe error summaries in logs.
- Restore drill evidence exists before launch.
- Background-job failures trigger alerts and expose safe diagnostics.
- Sensitive-log review passes before production.

## Risks

- Observability may lag behind feature work.
- Logs may leak sensitive data without redaction tests.
- Backups do not prove recoverability without restore drills.

## Follow-ups

- Define log-redaction test cases.
- Create backup/restore runbook and evidence template.
- Configure alerts for API, worker, provider, backup, and DB failures.
