# ARD-0020 — Server-Side Cache Policy

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Performance / Data Consistency
**Required Before:** Milestone 1
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS allows browser PWA caching and optional server-side ephemeral caching. PostgreSQL remains authoritative, and offline cache is read-only.

## Decision

Do not add Redis or another server-side cache initially unless measured performance requires it.

Allowed caches:

- Browser PWA app-shell cache.
- User-scoped read-only recent-record cache.
- Short-TTL in-process caches for low-risk configuration, invalidated on relevant admin changes.

Initially disallowed:

- Server-side cache as source of truth.
- Cache-backed stock quantities or invoice balances.
- Permission caches without invalidation.
- Tenant lifecycle status caches without freshness control.

## Consequences

This reduces infrastructure and consistency risk while keeping PostgreSQL authoritative. Redis may be added later only with production metrics justification.

## Validation

- Performance tests determine whether server-side cache is needed.
- Security tests verify offline/user cache clears on logout or session revocation.
- Permission and tenant-status updates are reflected without stale unsafe access.

## Risks

- No server-side cache may expose load-related performance issues.
- In-process caches may become stale across API instances.
- Incorrect auth/status caching can cause unsafe access.

## Follow-ups

- Measure API/report performance before adding Redis.
- Define in-process cache TTLs and invalidation triggers.
- Add stale-permission/status tests if caching is introduced.
