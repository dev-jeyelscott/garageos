# ARD-0014 — Idempotency and Optimistic Locking

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Reliability / Data Integrity
**Required Before:** Milestone 2
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

Critical writes can be retried or duplicated on unstable networks, especially payments, receipts, refunds, inventory movements, exports, and tenant deletion.

## Decision

Implement a shared idempotency service for critical writes.

Idempotency keys are scoped by:

- Tenant.
- Authenticated user.
- Endpoint/action.
- Request intent hash.
- Idempotency key hash.

Use `lock_version` optimistic locking for documented mutable records. Use row locks for high-contention financial and inventory commands when optimistic locking is insufficient.

Initial `If-Match` support uses raw `lock_version`. An ETag wrapper may be added later without changing service behavior.

## Consequences

This prevents duplicate side effects for retryable critical writes, but adds request hashing, replay handling, retention, and cleanup complexity.

## Validation

- Same key + same intent returns replayed response.
- Same key + different intent returns conflict.
- Concurrency tests prove no overpayment, over-refund, overbilling, over-reservation, double FIFO consumption, or duplicate receipts.

## Risks

- Bad intent hashing may replay incorrect responses or block valid retries.
- Early cleanup may break safe retries.
- Optimistic locking alone may not protect high-contention financial/inventory workflows.

## Follow-ups

- Build idempotency table and service.
- Define retention and cleanup policy.
- Add concurrency tests for all critical writes.
