# ARD-0002 — Repository Model and TypeScript Workspace

**Status:** Accepted
**Date:** 2026-06-24
**Owner:** Engineering
**Type:** Repository / Tooling
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA plan, RTM, user stories, permission matrix, UX map, tech stack, roadmap, architecture records

## Context

GarageOS uses TypeScript across the PWA and backend. Shared contracts, utilities, validation, API clients, and tests need consistent workspace management.

## Decision

Use a TypeScript monorepo managed with `pnpm` workspaces.

```text
apps/
  web/
  api/
  worker/
  scheduler/
packages/
  shared/
  api-client/
  config/
  test-utils/
docs/
  adr/
  runbooks/
  api/
```

## Consequences

- Improves shared type and API contract consistency.
- Enables coordinated CI across apps and packages.
- Requires strict package boundaries to avoid coupling.

## Validation

- CI runs install, lint, typecheck, tests, and migration validation.
- Packages have no circular dependencies.
- Shared packages contain only cross-cutting primitives or generated/shared contracts.

## Risks

- Shared packages may collect domain logic.
- Workspace boundaries may weaken over time.
- Poor CI scoping may block all apps unnecessarily.

## Follow-ups

- Add dependency-boundary linting or review rules.
- Document package ownership and allowed dependencies.
- Add workspace bootstrap instructions to README.
