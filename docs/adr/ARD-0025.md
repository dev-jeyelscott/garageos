# ARD-0025 — Tenant Duplicate Approval Schema

**Status:** Accepted
**Date:** 2026-06-27
**Decision Owner:** Engineering
**Decision Type:** Database / API
**Source Alignment:** Requirements, user stories, schema, API contracts, permissions

## Context

GarageOS must block duplicate non-deleted tenants with the same normalized business name and shop email unless a platform admin explicitly approves the duplicate.

The current unique index blocks all non-deleted duplicates:

```sql
create unique index ux_tenants_active_business_email
on tenants(normalized_business_name, normalized_shop_email)
where status <> 'deleted';
```

This enforces safe default blocking but prevents the documented platform-admin approval exception.

## Decision

Do not implement approved duplicate tenant creation until the schema supports explicit approval.

Rules:

- Duplicate prevention remains default.
- Owner signup duplicates remain blocked.
- Duplicate approval applies only to platform-admin tenant creation.
- Approved duplicates require platform-admin actor, reason, and timestamp.
- Duplicate detection, approval, tenant creation, and subscription assignment must be audited.

## Preferred Minimal Schema

Add nullable approval fields to `tenants`:

```sql
duplicate_approved_at timestamptz null,
duplicate_approved_by_platform_admin_user_id uuid null,
duplicate_approval_reason text null
```

Replace the current index:

```sql
drop index if exists ux_tenants_active_business_email;

create unique index ux_tenants_unapproved_business_email
on tenants(normalized_business_name, normalized_shop_email)
where status <> 'deleted'
  and duplicate_approved_at is null;
```

Add metadata completeness constraint:

```sql
alter table tenants
add constraint chk_tenants_duplicate_approval_complete
check (
  (
    duplicate_approved_at is null
    and duplicate_approved_by_platform_admin_user_id is null
    and duplicate_approval_reason is null
  )
  or
  (
    duplicate_approved_at is not null
    and duplicate_approved_by_platform_admin_user_id is not null
    and duplicate_approval_reason is not null
    and length(trim(duplicate_approval_reason)) > 0
  )
);
```

Service validation must still verify the approver is a platform admin with tenant creation permission.

## API Direction

Platform tenant creation may later accept:

```json
{
  "approve_duplicate": true,
  "duplicate_approval_reason": "Separate legal branch using same email during migration."
}
```

Validation:

- Duplicate without approval returns `409 duplicate_resource`.
- Approval without nonblank reason returns `422 validation_failed`.
- Non-platform or unauthorized actor returns `403 forbidden`.
- Valid approval creates tenant with approval metadata.
- Owner signup never accepts duplicate approval.

## Migration / Code Plan

1. Add nullable approval columns.
2. Drop old duplicate index.
3. Add unapproved-duplicate unique index.
4. Add approval completeness constraint.
5. Add DTO fields for platform tenant creation only.
6. Add service validation and audit logging.
7. Map duplicate constraint violations to `409 duplicate_resource`.

## Validation

Tests must cover:

- Duplicate without approval is blocked.
- Duplicate with valid approval is allowed.
- Blank approval reason is blocked.
- Owner signup duplicate remains blocked.
- Approved duplicate writes audit log.
- Concurrent unapproved duplicate creation still fails.

## Consequences

This preserves safe default duplicate blocking, enables the documented platform-admin exception, keeps owner signup safer, and makes approval metadata queryable.

Trade-offs:

- Database constraints cannot prove the actor is a platform admin.
- Service authorization and audit logging remain mandatory.
- Schema documentation must be updated after acceptance.

## Open Questions

- Should approval history become a separate table?
- Should approval apply to all non-deleted statuses or only operational statuses?
- Should multiple approval events be retained?

## Recommendation

Use the minimal column-based approach for Milestone 3 unless approval history is required as a first-class audit table.
