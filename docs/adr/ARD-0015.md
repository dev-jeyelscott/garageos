# ARD-0015 — Append-Only Immutability Protections

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Database Integrity / Security
**Required Before:** Milestone 1 baseline; production hardening before launch
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS requires financial records, receipts, refunds, inventory ledgers, FIFO records, and audit logs to be immutable or correction-only. Immutability trigger timing must be decided before implementation.

## Decision

Use two-level append-only protection:

1. **Application level from first implementation:** command services expose no direct update/delete paths for immutable records.
2. **Database level before production launch:** triggers block updates/deletes on immutable tables, except documented correction/status fields.

Critical immutable tables include audit logs, receipts, issued invoice records/lines where applicable, payments, refunds, inventory ledger entries, FIFO allocations/consumptions, and background side-effect history.

## Consequences

This protects audit, financial, and inventory integrity, but requires explicit correction workflows and carefully scoped trigger exceptions.

## Validation

- Schema tests reject unauthorized update/delete attempts.
- API tests prove corrections, voids, and refunds create new records or allowed status changes instead of mutating issued records.
- Security review confirms privileged users cannot bypass audit logging through application APIs.

## Risks

- Triggers may block valid corrections if exceptions are incomplete.
- Late trigger implementation may expose invalid mutation paths.
- Superuser or migration access can bypass protections without process controls.

## Follow-ups

- List immutable tables and allowed correction fields.
- Create trigger migration templates.
- Add production-hardening checklist before launch.
