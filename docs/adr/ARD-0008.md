# ARD-0008 — SQL-Visible Query Layer

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Database Access
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA, RTM, user stories, permissions, UX, tech stack, roadmap, ARDs

## Context

Critical GarageOS workflows require visible SQL semantics for tenant/branch scoping, transactions, locks, optimistic locking, FIFO allocation, idempotency, reports, and concurrency safety.

## Decision

Use **Kysely with `node-postgres`** as the primary query layer.

Allow raw SQL for:

- Row locks.
- CTEs.
- Reports.
- FIFO and financial queries.
- Migration-time data fixes.
- Performance-critical paths.

Repository rules:

- Tenant-owned queries must include `tenant_id`.
- Branch-specific queries must include `tenant_id` and `branch_id` where applicable.
- Critical commands must run in explicit transactions.
- Row locks and optimistic locks must be visible in repository code.

## Consequences

- Keeps SQL typed but transparent.
- Supports complex FIFO, financial, and reporting queries.
- Requires SQL and indexing discipline.

## Validation

- Run integration tests against real PostgreSQL via Testcontainers.
- Review high-volume, report, ledger, FIFO, and financial queries.
- Concurrency tests must prevent over-reservation, overpayment, overbilling, duplicate receipts, and duplicate numbers.

## Risks

- SQL, locking, and indexing mistakes remain possible.
- Raw SQL can reduce type safety if overused.
- Missing tenant/branch predicates can cause isolation defects.

## Follow-ups

- Create tenant/branch-scoped repository templates.
- Add Testcontainers PostgreSQL test harness.
- Document approved raw SQL use cases.
