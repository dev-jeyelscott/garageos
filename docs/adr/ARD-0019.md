# ARD-0019 — External Provider Adapter Strategy

**Status:** Accepted with provider selection deferred
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Integrations / Operations
**Required Before:** Milestone 10/13
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS needs providers for email, SMS, push, analytics, error monitoring, and storage, but source docs do not mandate vendors. Automated subscription payment collection remains out of scope.

## Decision

Use internal provider adapters:

```text
EmailProvider
SmsProvider
PushProvider
AnalyticsProvider
ErrorMonitoringProvider
StorageProvider
MalwareScanProvider optional/deferred by risk decision
```

Vendor selection is deferred to environment, cost, and compliance review. Domain logic must depend on adapters, not vendor SDKs.

Selection criteria:

- Target-market availability.
- Delivery logs/status callbacks where applicable.
- Secure secret management.
- Cost predictability.
- Sandbox support.
- Operational monitoring.
- Sensitive-payload log avoidance.

## Consequences

This keeps product logic vendor-neutral and allows provider replacement without scope change, but requires adapter contracts and fake providers for tests.

## Validation

- Integration tests use fake providers.
- Staging uses sandbox providers where available.
- Delivery attempts/failures appear in logs and background-job records without leaking secrets.

## Risks

- Deferred selection may delay Milestone 10/13 integration work.
- Adapter contracts may miss provider-specific statuses or failures.
- Secrets or payloads may leak without redaction tests.

## Follow-ups

- Select providers before their milestone gates.
- Add fake provider implementations.
- Define provider secret/payload redaction checks.
