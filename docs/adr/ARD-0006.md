# ARD-0006 — REST API Contract, Naming, and DTO Style

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** API
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA, RTM, user stories, permissions, UX, tech stack, roadmap, ARDs

## Context

GarageOS requires a predictable REST API with `/api/v1`, JSON envelopes, stable enum values, explicit workflow actions, and consistent error semantics.

## Decision

Use a REST-first API under `/api/v1`.

API payloads must use `snake_case` to align with schema columns, enum values, exports, and API contracts.

Workflow transitions must use explicit action endpoints, for example:

```text
POST /invoices/{id}/issue
POST /invoices/{id}/payments
POST /payments/{id}/refunds
POST /inventory-transfers/{id}/send
POST /inventory-transfers/{id}/receive
```

Do not expose arbitrary status PATCHing for state machines.

## Consequences

- Keeps frontend, backend, and QA behavior predictable.
- Protects workflow invariants.
- Frontend may map to camelCase internally, but API payloads remain `snake_case`.

## Validation

- Contract tests verify response envelopes, error envelopes, status codes, error codes, correlation IDs, and `snake_case` payloads.
- Workflow tests verify unsupported status changes are blocked.

## Risks

- New action endpoints may drift from the API contract.
- snake_case/camelCase mapping may add frontend friction.
- Inconsistent envelopes would weaken QA determinism.

## Follow-ups

- Define controller conventions and an example endpoint.
- Add envelope, error, and idempotency contract tests.
- Generate OpenAPI after the API skeleton exists.
