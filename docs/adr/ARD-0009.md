# ARD-0009 — Migration Tool and Schema Drift Control

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Database Migration
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA, RTM, user stories, permissions, UX, tech stack, roadmap, ARDs

## Context

`database-schema.md` is the canonical physical schema blueprint. Migrations must be versioned, CI-validated, and forward-only in production.

## Decision

Use **node-pg-migrate** as the canonical migration tool.

Allow raw SQL migration bodies for PostgreSQL-specific features:

- Constraints.
- Indexes.
- Triggers.
- RLS policies.
- Advanced database features.

Migration rules:

- Production migrations are forward-only.
- Seeds must be idempotent.
- Destructive changes require backup and rollout plans.
- Constraint/index migrations require tests where applicable.
- CI must validate migrations against a clean PostgreSQL database.

## Consequences

- Supports PostgreSQL-specific schema features.
- Avoids ORM-generated schema drift.
- Requires strict migration review.

## Validation

- CI applies all migrations from zero to latest.
- CI checks schema drift for tables, enums/check constraints, indexes, and seed data.
- Non-production rollback rehearsal is documented where practical.

## Risks

- Migrations may drift from `database-schema.md`.
- Forward-only production migrations require strong recovery planning.
- Destructive migrations risk data loss without backup evidence.

## Follow-ups

- Create migration folder and naming convention.
- Add clean-database migration validation in CI.
- Document destructive migration backup/rollout requirements.
