# ARD-0017 — Reporting/Search Read Model Refresh Strategy

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Reporting / Performance
**Required Before:** Milestone 6 baseline; full before Milestone 12
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

Reports, dashboards, and search must scale without overloading transactional workflows. Source docs allow PostgreSQL-backed materialized or maintained aggregate read models.

## Decision

Use rebuildable PostgreSQL read models:

- Search documents updated by transactional events/outbox where practical.
- Dashboard snapshots refreshed by background jobs.
- Report summaries refreshed by schedule or events based on report type.
- Transactional source tables remain authoritative.

Initial approach:

- Maintained aggregate tables for high-frequency dashboard metrics.
- Search tables with PostgreSQL FTS/trigram indexes for documented searchable entities.
- Async report/export jobs for large outputs.

## Consequences

This improves search/report/dashboard performance, but introduces freshness, reconciliation, and rebuild requirements.

## Validation

- Formula tests compare read models against transactional source truth.
- Rebuild jobs regenerate read models from source tables.
- Performance tests cover dashboard, search, ledger pagination, reports, and exports.

## Risks

- Read models may become stale or inconsistent.
- Report formulas may drift from financial rules.
- Search may underperform without query-plan review.

## Follow-ups

- Define freshness targets per read model.
- Design rebuild and reconciliation jobs.
- Add report formula verification fixtures.
