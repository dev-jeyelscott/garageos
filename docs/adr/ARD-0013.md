# ARD-0013 — PWA Token Transport and Session Security

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Authentication / Security
**Required Before:** Milestone 2
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS requires 15-minute access-token expiry, refresh-token rotation, remember-me sessions up to 30 days, session revocation, and rate limits. Token transport is an implementation decision.

## Decision

Use:

- Short-lived access tokens held in PWA memory by default.
- Rotating refresh tokens in `HttpOnly`, `Secure`, `SameSite=Lax` cookies for same-site deployments.
- CSRF protection for cookie-backed refresh/logout flows.
- Hashed refresh tokens in the database.
- Immediate refresh-session revocation on logout-all, user deactivation, and admin-triggered password reset.

For cross-site app/API deployments, replace or revise cookie and CSRF settings through a superseding ARD before production.

## Consequences

This reduces persistent token exposure to XSS, but requires robust refresh-on-reload handling and CSRF protection for cookie-backed flows.

## Validation

- Security tests cover refresh rotation, reuse detection, session revocation, deactivated-user lockout, password-reset revocation, and token redaction.
- API tests cover access-token expiry, logout, and logout-all behavior.

## Risks

- Cookie refresh flows are unsafe without CSRF controls.
- In-memory access tokens require reliable reload recovery.
- Cross-origin deployment may require revised cookie/security settings.

## Follow-ups

- Confirm same-site or cross-site deployment model.
- Implement CSRF strategy for refresh/logout.
- Add log/error token-redaction tests.
