# ARD-0001 — Architecture Record Governance

**Status:** Accepted
**Date:** 2026-06-24
**Owner:** Engineering
**Type:** Governance
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA plan, RTM, user stories, permission matrix, UX map, tech stack, roadmap, architecture records

## Context

GarageOS requires documented architecture decisions for cross-cutting technical choices that affect security, data integrity, operations, and cost.

## Decision

Create `/docs/adr/` and record high-impact architecture decisions using this format:

```text
# ARD-XXXX — Title
Status:
Date:
Decision Owner:
Source Alignment:
Context:
Decision:
Consequences:
Validation:
Risks:
Follow-ups:
```

Accepted ARDs are binding implementation guidance unless superseded by a later accepted ARD. Product behavior remains governed by the PRD and approved source documents.

## Consequences

- Reduces undocumented technical drift.
- Improves auditability of engineering decisions.
- Adds documentation overhead for major technical changes.

## Validation

- Review checklist or CI confirms high-impact changes reference an ARD.
- Feature tickets include relevant PRD, RTM, API, schema, permission, UX, QA, and ARD references.

## Risks

- ARDs may become stale if not enforced.
- Engineers may skip ARD updates under delivery pressure.
- Conflicting ARDs can cause drift if supersession is unclear.

## Follow-ups

- Add ARD reference checks to the PR template or review checklist.
- Define when to supersede versus edit an ARD.
- Keep the ADR inventory updated.
