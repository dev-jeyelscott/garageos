# ARD-0007 — Validation Strategy

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** API / Backend
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA, RTM, user stories, permissions, UX, tech stack, roadmap, ARDs

## Context

GarageOS requires deterministic validation errors, stable DTO behavior, precision checks for money and quantity, and clear blocked-action messages.

## Decision

Use **Zod schemas** for request validation and shared schema-derived typing where practical, integrated through NestJS pipes.

Keep DTO shape validation separate from service-layer business validation.

Validation layers:

1. Request DTO/schema validation.
2. Authorization and tenant/branch/status/plan guards.
3. Business-rule validation in command services.
4. Database constraints as final invariant protection.

## Consequences

- Keeps validation aligned with API contracts.
- Avoids relying on frontend validation.
- Requires discipline to avoid putting business rules in DTO schemas.

## Validation

- Contract tests verify `422 validation_failed` and field-level errors.
- Business-rule tests verify domain errors such as inventory insufficiency, overpayment, over-refund, and blocked workflow transitions.

## Risks

- Business rules may be duplicated across schemas and services.
- Schemas may drift from database constraints.
- Overly strict validation may block valid workflows.

## Follow-ups

- Define NestJS validation pipe integration.
- Create shared validation and business error-code mapping.
- Add schema/API enum drift tests.
