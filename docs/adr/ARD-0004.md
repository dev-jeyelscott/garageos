# ARD-0004 — UX Design System Strategy

**Status:** Accepted
**Date:** 2026-06-24
**Owner:** Engineering
**Type:** UX / UI
**Required Before:** Milestone 1
**Source Alignment:** PRD, database design/schema, architecture, API contracts, QA plan, RTM, user stories, permission matrix, UX map, tech stack, roadmap, architecture records

## Context

The UX map defines screen inventory, but not pixel-level wireframes. GarageOS needs a design-system baseline without adding undocumented scope.

## Decision

Use **Tailwind CSS** with a **shadcn/ui-style component foundation** built on accessible primitives.

Create low-fidelity wireframes before detailed UI for high-frequency workflows.

Required baseline patterns:

- Mobile navigation shell
- Tenant status banners
- Branch selector/indicator
- Permission-aware actions
- Offline read-only indicator
- Workflow confirmation modals
- Reason fields for audited/corrective actions
- Form validation and conflict states
- Background job status components

## Consequences

- Speeds consistent UI implementation.
- Avoids heavy design-system overhead early.
- Requires disciplined component reuse and accessibility review.

## Validation

- Components pass practical accessibility checks.
- Screens map only to documented route groups.
- QA covers permission, lifecycle, offline, empty, loading, validation, and conflict states.

## Risks

- Components may drift without ownership.
- Accessibility can regress if primitives are customized poorly.
- Wireframes may imply undocumented workflows if not traced to the UX map.

## Follow-ups

- Create low-fidelity wireframe backlog from `ux-sreen-map.md`.
- Define reusable form, modal, banner, and job-status components.
- Add accessibility checks to frontend review.
