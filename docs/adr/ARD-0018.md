# ARD-0018 — File Upload, Download, and Object Storage

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Files / Security
**Required Before:** Milestone 11
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS requires private tenant-scoped files, signed URLs, attachments, tenant exports, deletion lifecycle handling, and no permanent public tenant file URLs.

## Decision

Use private S3-compatible object storage with direct signed upload/download URLs.

Flow:

1. Client requests upload intent.
2. API validates tenant, branch, permission, tenant status, plan/status restrictions, and file type/size rules.
3. API creates file metadata in pending/active state.
4. API returns a short-lived signed upload URL.
5. API or worker finalizes metadata after upload confirmation.
6. Downloads require API authorization before signed URL generation.

The provider is an environment decision if it supports S3 compatibility, private buckets, signed URLs, lifecycle policies, and export packages.

## Consequences

This reduces backend bandwidth and preserves private file access control, but requires abandoned-upload cleanup.

## Validation

- Security tests block unauthorized tenant/branch signed URL generation.
- Upload tests block uploads in read-only, suspended, or offline states.
- File deletion/export tests verify retention and tenant-deletion rules.

## Risks

- Long-lived or logged signed URLs can leak access.
- Abandoned uploads may leave orphan metadata or objects.
- S3-compatible vendors may differ in behavior.

## Follow-ups

- Select object-storage provider before Milestone 11.
- Define file type/size rules and abandoned-upload cleanup.
- Add signed URL authorization tests.
