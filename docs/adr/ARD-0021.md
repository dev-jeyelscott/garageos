# ARD-0021 — Append-Only Table Partitioning Threshold

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Database Performance
**Required Before:** Milestone 13 hardening
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

Append-only tables such as audit logs, inventory ledgers, outbox events, background jobs, notification attempts, and lifecycle events can grow quickly. Source docs recommend partition-ready designs.

## Decision

Make high-volume append-only tables partition-ready from the first migration, but defer physical partitioning until metrics justify it.

Review partitioning when any condition occurs:

- Table exceeds **10M rows**.
- P95 indexed list/report query latency exceeds targets.
- Table/index bloat harms vacuum or autovacuum.
- Export/report workloads create sustained operational pressure.

Default strategy when triggered:

- Monthly time-based partitions for audit/log/job/notification-style tables.
- Tenant/date-aware partition indexes.
- Retention/archive policy aligned to PRD retention rules.

## Consequences

This avoids premature partition complexity while preserving migration options, but requires growth, latency, and bloat monitoring.

## Validation

- Migrations avoid choices that block future partitioning.
- Monitoring tracks row counts, index bloat, autovacuum lag, and P95 query latency.
- Partition rehearsal runs in staging before production activation.

## Risks

- Late partitioning may require complex migration.
- Delayed action can harm vacuum, backups, and query latency.
- Poor partition strategy can conflict with retention or deletion rules.

## Follow-ups

- Track row-count and index-bloat metrics.
- Document partition rehearsal procedure.
- Revisit thresholds during Milestone 13 hardening.
