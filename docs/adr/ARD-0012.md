# ARD-0012 — Tenant/Branch Isolation and RLS Timing

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Security / Database
**Required Before:** Milestone 1
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

Tenant isolation is foundational. GarageOS requires repository/service scoping from day one, with PostgreSQL RLS added as defense-in-depth before production where safely testable.

## Decision

Use layered isolation:

1. Resolve tenant context from session.
2. Enforce tenant status.
3. Enforce permissions.
4. Enforce branch access.
5. Require `tenant_id` and applicable `branch_id` in repositories.
6. Protect invariants with database keys, indexes, and constraints.
7. Add PostgreSQL RLS after repository policies stabilize, before production launch, for high-risk tenant-owned tables.

RLS must never be the only isolation layer.

## Consequences

This reduces cross-tenant leakage risk while avoiding premature, untested RLS blocking development. Production readiness requires dedicated RLS fixtures and staging validation.

## Validation

- Cross-tenant read/write tests for customers, motorcycles, products, invoices, reports, files, and exports.
- Branch access tests for assigned and unassigned branch records.
- Staging RLS policy tests before production enablement.

## Risks

- Repository scoping bugs may leak data before RLS is enabled.
- Poor RLS policies may block valid queries.
- Support access may bypass safeguards if not explicitly tested.

## Follow-ups

- Add tenant/branch repository fixtures.
- Prototype RLS on high-risk tables in staging.
- Test support access to prove no silent impersonation.
