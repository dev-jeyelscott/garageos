# ARD-0016 — Background Job Locking and Outbox Processing

**Status:** Accepted
**Date:** 2026-06-24
**Decision Owner:** Engineering
**Decision Type:** Background Processing
**Required Before:** Milestone 2
**Source Alignment:** Core PRD, database, architecture, API, QA, RTM, user stories, permissions, UX, tech stack, roadmap, and ARD docs

## Context

GarageOS needs retry-safe background processing for reminders, notifications, lifecycle evaluation, exports, file retention, provider delivery, outbox events, and deletion jobs. PostgreSQL-backed jobs/outbox are the initial architecture.

## Decision

Use PostgreSQL-backed `background_jobs` and `outbox_events` as the initial queue/outbox mechanism.

Worker claim query:

```sql
select *
from background_jobs
where status = 'queued'
  and run_at <= now()
order by priority desc, run_at asc, created_at asc
for update skip locked
limit :batch_size;
```

Rules:

- Claim jobs inside transactions.
- Track attempts, max attempts, last error summary, next retry time, and dead-letter status.
- Protect irreversible side effects with provider idempotency keys or internal side-effect records.
- Emit structured logs, metrics, and failure alerts.

## Consequences

This keeps enqueueing transactional and avoids extra queue infrastructure initially. A dedicated queue should be considered only after production evidence shows PostgreSQL-backed jobs are insufficient.

## Validation

- Concurrent worker tests prove no duplicate claims.
- Retry tests prove safe reprocessing and dead-letter behavior.
- Operational tests prove job status visibility and alerting.

## Risks

- PostgreSQL-backed queue may bottleneck at scale.
- Incomplete side-effect idempotency may duplicate provider actions.
- Long-running jobs may hold locks or starve smaller jobs if batching is poor.

## Follow-ups

- Create `background_jobs` and `outbox_events` repository patterns.
- Add claim, retry, and dead-letter tests.
- Define external queue extraction criteria.
