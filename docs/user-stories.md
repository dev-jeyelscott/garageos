# GarageOS User Stories

**Source:** `requirements.md`, `database-design.md`, `database-schema.md`, `architecture.md`, `api-contracts.md`  
**Purpose:** Low-token backlog index retaining all epics, US IDs, core acceptance intent, and source references. Use full source docs for complete field-level wording.

## Non-Negotiable Rules

- PRD wins; database/schema govern persistence; architecture governs structure/NFRs; API governs routes, envelopes, auth, idempotency, and errors.
- Global AC: tenant/branch scoping; permission + branch + lifecycle gates; no cross-tenant access; critical writes idempotent/locked; explicit workflow transitions; audit critical actions; immutable/correction-only financial/receipt/refund/ledger/FIFO/audit records; offline read-only; standard API envelopes.
- Exclusions: native apps, offline writes/sync, GL/accounting, payroll, BIR filing, marketplace, customer portal, loyalty, packages, AI/custom BI, auto subscription charging, standalone POS, 2FA.

## Compact Backlog

### E1 — Platform Administration, Tenant Lifecycle, and Subscriptions

- **US-001 Create platform-managed tenants** — can create a tenant; can assign plan, start date, and expiration date before activation; can create or invite the shop owner; Duplicate active tenant using the same normalized business/email blocked unless explicitly approved by platform admin. _Src: PRD 4.1, 4.2, 4.11, 4.12; API 9.2; Sch 5._
- **US-002 Support owner signup tenant creation** — signup requires email verification before ops access; signup creates a tenant in `pending_setup`; assigns the configured default plan and default subscription duration; Signup blocked if default plan or default subscription duration is not configured. _Src: PRD 4.2, 4.11, 4.12; API 9.1, 9.3._
- **US-003 Enforce tenant subscription lifecycle** — Before expiration, tenant status is `active` with full permission-based access; Day 1–14 after expiration sets `grace_period` with full access and renewal warnings; Day 15–30 sets `read_only` and blocks operational writes; Day 31–60 sets `suspended`, allowing only Owner renewal/export access. _Src: PRD 4.3–4.8, 4.12; Arch 10.4; API 5.4._
- **US-004 Renew subscription through external payment confirmation** — Renewal action doesn’t process payment in GarageOS; Renewal action shows renewal instructions or submits a renewal request; Tenant returns to `active` only after platform admin confirms external payment and updates subscription expiration date or override; If renewal occurs before permanent deletion completes, tenant is restored to `active` immediately after platform admin confirmation. _Src: PRD 4.4, 4.12, 5.5; API 9.3._
- **US-005 Manage subscription plans and plan limits** — supports exactly Basic, Mid, and High standard plans; can configure plan limits; can configure one default active plan for owner signup; Plan limits include active branches, notification channels, customer reminders, branch reports, and advanced reports. _Src: PRD 5.1–5.5; Sch 5, 23.1; API 9.2._
- **US-006 Enforce branch limits by plan** — Active branch count is checked before creating or reactivating a branch; creation blocked when active branch count reaches the effective plan limit; Inactive branches don’t count against active branch limit; Blocked attempts show a clear upgrade prompt. _Src: PRD 5.3, 9.5; API 9.4._
- **US-007 Manage audited platform support access** — Support access requires platform admin, tenant, access reason, start timestamp, expiration/end timestamp, access mode, and audit log; Default support access mode is `read_only`; `write_allowed` is explicitly selected and permission-controlled; Platform admins no silently impersonate tenant users. _Src: PRD 4.10; Arch 22.4; API 9.2._

### E2 — Authentication and Account Management

- **US-008 Log in, maintain sessions, and log out** — can log in with email and password; Access tokens expire within 15 minutes; Refresh tokens rotate; Remember-me sessions can last up to 30 days. _Src: PRD 6.1, 6.6, 6.9; API 9.1._
- **US-009 Verify email before operational access** — Before verification, user can only access verification screen, resend verification action, and logout; Operational access blocked until email is verified; Email verification token is single-use and stored securely as a hash; Verification is reflected in session state. _Src: PRD 6.3; API 9.1; Sch 6._
- **US-010 Reset forgotten password securely** — Password reset requests are rate-limited to 3 per account per hour; Password reset links expire after 30 minutes; Password reset tokens are single-use and stored as hashes; Existing sessions are revoked after successful admin-triggered reset. _Src: PRD 6.4, 6.5, 6.8, 10.4; API 9.1, 9.5._
- **US-011 Protect login from brute-force attempts** — rate-limits 5 failed login attempts within 15 minutes per account and per IP address; Excess attempts blocked for 15 minutes; Forgot-password requests remain available subject to password reset rate limits; Lockout events audited. _Src: PRD 6.7, 33.6; API 12.3._
- **US-012 Change user email safely** — Email change requires own confirmation or authorized admin action; New normalized email is globally unique among active users; New email is verified before ops access continues; Active sessions are revoked after email change confirmation. _Src: PRD 6.10; API 9.1, 9.5._
- **US-013 Bootstrap first platform admin securely** — Bootstrap is disabled after first platform admin is created; Bootstrap requires infrastructure-level operator access; Bootstrap requires strong temporary credential or invite token; Bootstrap action writes platform audit log. _Src: PRD 6.2.1._

### E3 — Roles, Permissions, Employees, and Branch Access

- **US-014 Seed tenant role templates** — Tenant setup seeds Shop Owner, Manager, Service Advisor, Mechanic, Cashier, and Inventory Clerk roles; Default role templates belong to the tenant after creation; role no be deleted, deactivated, or modified in a way that removes required owner capabilities; Role permission changes audited. _Src: PRD 7.1–7.13; Sch 23.2; API 9.5._
- **US-015 Resolve additive permissions across multiple roles** — Users may have multiple roles; Effective permissions are the union of all active assigned role permissions; Explicit deny permissions are not supported; Permission changes take effect immediately for users assigned to the edited role. _Src: PRD 7.8, 7.9, 7.13; API 5.5._
- **US-016 Create and invite employees** — Public self-registration into an existing tenant as employee blocked; invitations are single-use and expire after 7 days; Invitation is scoped to exactly one tenant; Invitation includes role and branch configuration or requires completion before ops access. _Src: PRD 6.2.2, 10.1–10.2; API 9.5._
- **US-017 Deactivate employees safely** — Deactivated employee no log in; Active sessions are revoked; Open job orders assigned to deactivated employee are flagged for reassignment; Last active Owner no be deactivated or demoted. _Src: PRD 6.9, 7.2, 10.3; API 9.5._
- **US-018 Reactivate employees with current validations** — Reactivation requires `users.update` permission; belong to the same tenant; have at least one active branch assignment unless tenant-wide branch access is enabled; Global active email uniqueness is re-checked. _Src: PRD 10.5; API 9.5._
- **US-019 Assign branch access to employees** — have at least one active branch assignment unless tenant-wide branch access is enabled; Branch-specific records require assignment to the branch or tenant-wide branch access; Employees assigned only to a deactivated branch are flagged for reassignment; access is enforced for all branch-scoped operational records. _Src: PRD 7.10, 9.4, 10.2; Arch 10.3; API 5.3._
- **US-020 Manage custom roles** — Custom role belongs to exactly one tenant; Role name is unique per tenant among active roles; Role contains one or more action-level permissions; Role can be edited by users with `roles.update` permission. _Src: PRD 7.8, 7.12, 7.13; API 9.5._

### E4 — Shop Onboarding, Settings, and Branch Management

- **US-021 Complete shop onboarding** — Onboarding requires shop profile, at least one active branch, invoice prefix, tax profile, country, business timezone, default currency, and at least one active Shop Owner; Pending setup users can access only onboarding, profile setup, subscription information, password management, and logout; Operational modules remain blocked until onboarding completion; Tenant moves from `pending_setup` to `active` only when onboarding is complete and subscription plan and expiration date are effective. _Src: PRD 4.11, 8.1–8.2; API 9.3._
- **US-022 Configure shop profile and settings** — Required setup fields validate per rules; prefix match `^[A-Z0-9]{2,10}-$` and end with dash; prefix is immutable after onboarding completion; Tax profile, tax mode, and VAT rate changes affect only future invoices. _Src: PRD 8.2–8.4, 24, 36.2; API 9.3._
- **US-023 Create and update branches** — requires unique active name per tenant, address, contact number, business hours, and status; creation respects active branch plan limit; records are tenant-scoped; records use active/inactive status. _Src: PRD 9.1–9.2; API 9.4; Sch 7._
- **US-024 Deactivate branches safely** — Last active branch no be deactivated; deactivation blocked when open job orders, open purchase orders, pending/in-transit transfers, active reservations, non-zero on-hand inventory, or unposted stock-affecting records exist; Deactivated branch remains visible in historical reports; Deactivated branch is not selectable for new job orders, invoices, purchases, or transfers. _Src: PRD 9.3; API 9.4._
- **US-025 Reactivate branches with plan validation** — Reactivation requires `branches.reactivate` permission; belong to same tenant; Effective active branch limit is re-checked; Reactivation blocked if active branch limit would be exceeded. _Src: PRD 9.5; API 9.4._

### E5 — Customer and Motorcycle Management

- **US-026 Create and search customers** — requires name and at least one contact method: mobile number or email; Customers are tenant-wide and visible to authorized users across branches; Duplicate detection warns on normalized mobile, lowercased email, and similar name; Duplicate warnings don’t automatically merge records. _Src: PRD 11.1–11.5; API 9.6; Sch 8._
- **US-027 Merge duplicate customers** — Merge requires `customers.merge` permission; selects one surviving customer; Service history, motorcycles, invoices, payments, reminders, and files are preserved; Duplicate customer is marked `merged` and stores surviving customer ID. _Src: PRD 11.6; API 9.6._
- **US-028 Soft delete and restore customers** — soft deletion blocked when customer has open job orders, unpaid invoices, active reminders, or active motorcycles not reassigned or soft deleted; Soft-deleted customer is not selectable for new motorcycles, job orders, estimates, invoices, or reminders; Restoration requires `customers.restore` permission; Restoration re-checks duplicate warning rules. _Src: PRD 11.7–11.8; API 9.6._
- **US-029 Create and manage motorcycles** — reference an active customer in same tenant; requires brand, model, and mileage; Provided identifiers are normalized and checked for duplicates per tenant; Exact duplicate active identifier values blocked. _Src: PRD 12.1–12.6; API 9.7; Sch 8._
- **US-030 Track and correct odometer readings** — Completed job order stores motorcycle mileage at service time; latest mileage is updated only when completed job order mileage is >= current latest mileage; Lower mileage correction blocked unless user has `motorcycles.update` permission and provides correction reason; Mileage corrections audited. _Src: PRD 12.7; API 9.7._
- **US-031 Soft delete and restore motorcycles** — soft deletion blocked when motorcycle has open job orders, active reminders, or unpaid invoices linked through active job orders; Soft-deleted motorcycle is not selectable for new job orders; Restoration requires `motorcycles.restore` permission; Restoration re-checks exact duplicate constraints for identifiers. _Src: PRD 12.8–12.9; API 9.7._

### E6 — Service Catalog and Estimates

- **US-032 Manage predefined services** — Predefined service requires unique active service name per tenant, starting price, variable price flag, and status; Price disclaimer required when variable price is true; Service deactivation blocked when referenced by open job orders or active draft/presented estimates; Deactivated service is not selectable for new job order or estimate lines. _Src: PRD 13.1–13.5; API 9.8; Sch 9._
- **US-033 Add custom service lines to job orders** — Custom service line belongs only to the job order where it is created; Custom service line doesn’t automatically create a reusable catalog service; Custom service line is invoiceable as labor or service revenue. _Src: PRD 13.4, 14.11.1; API 9.10._
- **US-034 Create and present estimates** — Estimate is created as `draft` with tenant, branch, customer, estimate number, creator, and timestamp; Estimate number follows `EST-YYYYMMDD-000001`, tenant-wide and daily reset; Estimate number is immutable and never reused; Presented estimates can expire after valid until date when not approved, converted, or cancelled. _Src: PRD 15.1–15.3; API 9.9._
- **US-035 Approve, cancel, and convert estimates** — Presented estimate can be approved only when customer approval method is recorded; Approval records approved customer name, method, timestamp, recording user, and optional attachment; Converted, cancelled, and expired estimates are final. _Src: PRD 15.3, 15.6; API 9.9._
- **US-036 Keep estimates non-financial and non-inventory-impacting** — Estimates don’t reserve inventory; Estimates don’t affect sales revenue; Estimates don’t affect inventory on-hand or reserved stock; Estimates don’t affect FIFO layers, tax reports, or financial reports. _Src: PRD 15.4–15.5; API 9.9._

### E7 — Job Orders and Mechanic Time Tracking

- **US-037 Create job orders** — requires active branch, active customer, motorcycle belonging to selected customer, service advisor, mileage at intake, customer concern, creator, and timestamp; number follows `JO-YYYYMMDD-000001`, tenant-wide and daily reset; number is immutable and unique per tenant; Initial status is `pending`. _Src: PRD 14.1–14.4; API 9.10; Sch 9._
- **US-038 Enforce job order status transitions** — supports exactly `pending`, `in_progress`, `waiting_for_parts`, `completed`, `released`, and `cancelled` API statuses; Transition from pending to in progress requires primary mechanic assigned; Waiting for parts requires authorized status change and reason; Completed to in progress correction requires correction permission and reason. _Src: PRD 14.4–14.7; API 9.10, 10.3._
- **US-039 Add and reserve job order parts** — belong to same tenant; Stock branch match job order branch; Available stock is sufficient; Reservation reduces available stock but not on-hand stock. _Src: PRD 14.9, 17.8; API 9.10._
- **US-040 Complete job orders with inventory consumption** — Completion consumes reserved parts only when moving from in progress or waiting for parts to completed; reduces branch on-hand stock and reserved stock; creates inventory ledger entries using `job_order_consumption`; consumes FIFO layers from oldest allocated stock first. _Src: PRD 14.10, 18.4–18.5; Arch 12, 14; API 9.10._
- **US-041 Release completed job orders** — Release is allowed when fully paid, no-charge with reason, or release-with-balance permission and reason; Fully paid release requires issued invoice exists, billable lines fully billed, and linked issued invoices have zero remaining collectible balance; Release with balance requires `job_orders.release_with_balance` permission and release reason; Released job orders are immutable except viewing, file viewing, and audit log viewing. _Src: PRD 14.8; API 9.10._
- **US-042 Cancel job orders safely** — Cancellation requires `job_orders.cancel` permission and reason; Cancellation releases all active inventory reservations linked to job order; Cancellation blocked when released; Cancellation blocked when paid invoice balances remain unresolved. _Src: PRD 14.7; API 9.10._
- **US-043 Track mechanic work sessions** — Mechanic work session requires tenant, branch, job order, mechanic, start time, and calculated active duration; match job order branch; A mechanic no have more than one unfinished session across all job orders in the tenant. _Src: PRD 16.1–16.5; API 9.11; Sch 9._
- **US-044 Calculate mechanic productivity** — Productivity uses number of assigned jobs; Productivity uses completed job orders with mechanic sessions; Productivity includes rework/correction count based on status rollbacks; Reports attribute work based on actual mechanic sessions, not only assignment. _Src: PRD 14.13, 16.6; Reports 29.4._

### E8 — Products, Inventory, FIFO, Adjustments, and Transfers

- **US-045 Manage product catalog and categories** — Products are tenant-wide; Stock balances are branch-scoped; requires name, SKU, category, unit of measure, default cost, selling price, reorder level, and status; SKU is unique per tenant across active and inactive products. _Src: PRD 17.1–17.4; API 9.12; Sch 10, 23.2._
- **US-046 Deactivate and reactivate products/categories safely** — deactivation blocked when product has non-zero on-hand stock, active reservations, open job orders, open purchase orders, or draft/pending/in-transit transfer references; Deactivated product is not selectable for new job order parts, purchases, transfers, or positive adjustments; reactivation requires SKU and barcode uniqueness; deactivation blocked when active products are assigned. _Src: PRD 17.3.1, 17.4.1; API 9.12._
- **US-047 View stock balances and low-stock alerts** — tracks on-hand, reserved, and available stock per product per branch; Available stock is calculated as on hand minus reserved; Normal transactions no make available stock negative; Low-stock alert triggers when available is <= reorder level. _Src: PRD 17.5, 17.9; API 9.12; Sch 10._
- **US-048 Preserve immutable inventory ledger** — Every stock-changing event creates an inventory ledger entry; Direct stock quantity updates without ledger entry are not allowed; Inventory ledger records tenant, branch, product, source transaction, quantity deltas, costs, and timestamp; Inventory reports read ledger and source records according to branch access. _Src: PRD 17.6–17.7; Sch 10; Arch 14._
- **US-049 Maintain FIFO cost layers and consumption** — FIFO layers are created by purchase receiving, positive adjustments, transfer receiving, void inventory reversal, and refund inventory reversal; FIFO consumption occurs through job order consumption, transfer out, transfer variance loss, negative adjustments, and supplier returns; FIFO consumes oldest available stock first; Stock valuation reports calculate value using remaining FIFO quantities and unit costs. _Src: PRD 18.1–18.6; Sch 10; Arch 14._
- **US-050 Create inventory adjustments with approval workflow** — Adjustment requires permission, reason, branch, product, and quantity difference or final counted quantity; Adjustments creating positive stock create FIFO layers; Adjustments decreasing stock consume FIFO layers; Adjustments no make on-hand lower than reserved. _Src: PRD 17.10–17.11; API 9.13; Sch 10.1._
- **US-051 Perform force adjustments only as exceptional corrections** — Force adjustment requires `inventory.force_adjust` permission and reason; Force adjustment audited; Force adjustment creates inventory ledger entries; Force adjustment preserves tenant and branch isolation. _Src: PRD 17.12; API 9.13._
- **US-052 Create and submit inventory transfers** — Transfers occur only between active branches same tenant; Source and destination branches is different; number follows `TR-YYYYMMDD-000001`, tenant-wide and daily reset; Draft transfer has no stock effect. _Src: PRD 19.1–19.5; API 9.14; Sch 10.2._
- **US-053 Send and receive inventory transfers with FIFO preservation** — Sending transfer requires sent quantity >0 and not exceeding reserved quantity; If sent quantity is lower than reserved quantity, unused reservation is released; Source on-hand and reserved stock decrease by sent quantity; Destination on-hand increases by received quantity. _Src: PRD 19.6–19.7; API 9.14._
- **US-054 Record transfer variance and in-transit cancellation** — If received quantity is lower than sent quantity, variance reason required; Variance creates `inventory_transfer_variance_loss` ledger entry; Missing quantity is not automatically added to destination stock; In-transit cancellation requires `inventory.transfer.cancel` permission. _Src: PRD 19.8–19.9; API 9.14._

### E9 — Suppliers, Purchases, Supplier Returns, and Accounts Payable

- **US-055 Manage suppliers** — requires unique active supplier name per tenant; can store contact person, mobile number, email, address, and notes; is tenant-wide; can be deactivated and reactivated. _Src: PRD 20.1–20.5; API 9.15; Sch 11._
- **US-056 Create and manage purchase orders** — order requires active branch, supplier, payment terms, order date, and product lines; order number follows `PO-YYYYMMDD-000001`, tenant-wide and daily reset; status follows lifecycle: draft, ordered, partially received, received, closed, cancelled; Open purchase orders block branch deactivation as documented. _Src: PRD 21.1–21.4, 21.8; API 9.16; Sch 11._
- **US-057 Receive purchased stock and create FIFO layers** — Receiving requires active purchase order and valid received quantities; Receiving increases branch on-hand stock; Receiving creates FIFO layers with received unit cost; Receiving creates inventory ledger entries using `purchase_receive`. _Src: PRD 21.5–21.6, 21.11; API 9.16; Sch 11._
- **US-058 Track supplier payments and accounts payable** — payments require supplier, amount >0, payment date, payment method, and creator; payment method uses payment method enum values when applicable; balance reflects credit purchases, supplier payments, supplier returns, and supplier credits per AP rules; payment creation is idempotent. _Src: PRD 21.6–21.7, 23.1.1; API 9.20; Sch 11._
- **US-059 Process supplier returns** — return requires branch, supplier, reason, returned products, and quantities; Posted supplier return consumes FIFO layers; Posted supplier return reduces branch stock; return updates supplier payable or creates supplier credit according to payment state. _Src: PRD 21.9–21.10; API 9.17; Sch 11._

### E10 — Invoices, Payments, Receipts, Refunds, AR, and Tax

- **US-060 Create draft invoices from job orders** — is connected to one or more job orders; can include service, labor, parts, and custom lines per invoice scope; number uses tenant invoice prefix and sequence rules; number is unique per tenant and immutable after creation. _Src: PRD 22.1–22.8; API 9.18; Sch 12._
- **US-061 Issue invoices with immutable billing data** — Issuing invoice finalizes billing allocations; Issued invoice records copied tax profile, tax mode, VAT rate, discounts, and calculated totals; Issued invoices affect accounts receivable and financial reports; Issued invoices no be edited directly except through cancellation, void, refund, or corrected invoice workflows. _Src: PRD 22.7–22.13; API 9.18; Arch 15._
- **US-062 Prevent invoice overbilling** — lines originating from job order lines create billing allocations; Billing allocation statuses follow lifecycle; Remaining billable amount or quantity subtracts reserved, final, and closed allocations; Concurrent invoice drafts no over-allocate source job order lines. _Src: PRD 22.3.1; Arch 15.1; API 4.6, 9.18._
- **US-063 Calculate discounts and tax correctly** — Invoice-level discount is allocated across invoice lines before tax calculation; Tax profile is one of `vat_registered`, `non_vat`, or `no_tax`; Issued invoice retains copied tax settings used at issuance; Tax changes affect only future invoices. _Src: PRD 22.11–22.13, 24; API 9.18._
- **US-064 Record payments and generate immutable receipts** — is recorded against one invoice; amount is >0; no exceed remaining collectible balance; Partial and split payments are supported. _Src: PRD 23.1–23.6; API 9.19; Arch 15.2._
- **US-065 Correct payments through void/refund workflows** — Original payments and receipts are not deleted or edited for correction; correction uses refund or void workflows; amount no exceed refundable amount; recalculates invoice balance and status. _Src: PRD 23.2.1, 23.7; API 9.19; Arch 15.3._
- **US-066 Reverse inventory during refund or void only when selected** — doesn’t automatically restore inventory; Authorized user may select inventory reversal when stock is returned; inventory reversal creates FIFO layer when stock returns to inventory; Void inventory reversal creates FIFO layer when stock returns to inventory. _Src: PRD 18.2, 23.8; API 9.19._
- **US-067 Track accounts receivable** — Issued invoices with remaining collectible balance appear in AR; Partial payments reduce invoice remaining balance; Refunds recalculate invoice balance/status; Release-with-balance job orders keep remaining invoice balance in AR. _Src: PRD 23.9, 29.7, 29.10; API 9.20._

### E11 — Expenses, Reminders, Notifications, Files, Reports, and Export

- **US-068 Manage expenses and categories** — requires fields including branch when applicable, category, amount, payment method, date, and status; editing follows editing rules; void requires permission and reason; category deactivation follows blocking rules. _Src: PRD 25.1–25.4; API 9.21; Sch 13._
- **US-069 Create customer reminders** — supports reminder categories; statuses include scheduled, due, sent, failed, and cancelled; Time-based reminder due logic follows tenant timezone; creation and updates respect tenant status and permissions. _Src: PRD 26.1–26.8; API 9.22; Sch 14._
- **US-070 Enforce reminder delivery channels by plan** — In-app, push, email, and SMS channels follow effective plan availability; Disabled channel actions blocked with required plan level shown. _Src: PRD 5.2, 5.4, 26.4–26.5; API 9.22._
- **US-071 Track reminder delivery and retries** — Each delivery attempt records attempt number, provider/channel, timestamp, result status, and failure reason when available; Retry behavior follows retry rules; Permanent failures such as missing contact details, blocked plan channel, invalid address, or invalid phone number are not retried automatically; After all retry attempts fail, channel delivery status is marked failed. _Src: PRD 26.9–26.10, 27.1; API 9.22._
- **US-072 Send internal notifications** — supports notification types including low stock, new job orders, assignments, service completion, payments, subscription alerts, transfer updates, purchase receiving updates, reminder due alerts, failed reminder delivery, employee deactivation, and role/permission changes; Recipients are determined by tenant, branch access, permissions, notification type, preferences, and plan channel availability; Users no enable channels blocked by the tenant plan; Delivery status is tracked per recipient and channel. _Src: PRD 27.1–27.4; API 9.23; Sch 14._
- **US-073 Upload and access files securely** — Supported attachment scopes include motorcycle photos, service photos, job order attachments, receipt attachments, warranty documents, expense receipts, estimate approval attachments, supplier documents, and purchase documents; Supported image types are JPG, JPEG, PNG, and WEBP; Unsupported file types blocked; Files are stored under tenant-scoped private paths. _Src: PRD 28.1–28.4; Arch 20; API 9.24._
- **US-074 Soft delete, restore, and retain files** — Files are soft deleted for 30 days before permanent deletion; Soft-deleted files are excluded from default attachment lists; Soft-deleted files remain accessible to platform admins for audit support; Files linked to financial records, receipts, invoices, or audit-relevant documents are not permanently deleted before tenant retention policy allows deletion. _Src: PRD 28.4–28.5; Arch 20; API 9.24._
- **US-075 View dashboard and reports** — Dashboard supports operational summary features; Reports include sales, service, inventory, customer, and financial reports; filtering respects user branch access; Basic, branch comparison, and advanced reports follow plan access rules. _Src: PRD 29.1–29.10; Arch 21; API 9.25._
- **US-076 Export reports** — Report export supports export formats; Report export respects permission, plan access, tenant isolation, and branch access; Large report exports run through background jobs; Export status is visible through safe job status metadata. _Src: PRD 28, 29.8; Arch 19, 21; API 9.25, 9.28._
- **US-077 Generate full tenant export** — Tenant export includes CSV data, JSON relationship data, audit log export, attachment manifest, optional attachment binaries, and README; Large exports run asynchronously; Export download links expire after 7 days; Export respects tenant lifecycle rules. _Src: PRD 31.1–31.2; Arch 20.4; API 9.26._
- **US-078 Enforce data retention and tenant deletion** — Tenant enters pending deletion Day 61 after expiration; Deletion scheduled date is Day 68 unless emergency extension is granted; Deletion warnings are sent Day 61, Day 65, and Day 67 when email contact is available; Deletion job permanently deletes eligible tenant production database records. _Src: PRD 4.8, 30.4, 31.3–31.4; API 9.26, 9.28; Sch 5._

### E12 — Audit, Offline PWA, Security, Observability, and Operations

- **US-079 Write immutable audit logs for critical actions** — capture required fields; Required audit coverage includes business, security, platform, support, and corrective actions; are immutable; Platform support access and support actions are audited. _Src: PRD 30.1–30.4; Sch 16; API 9.27._
- **US-080 Support offline shell and read-only cache** — PWA is installable and loads application shell without network connection; Offline state is clearly indicated; Recently viewed customers, motorcycles, job orders, and invoices may be cached; Cache is read-only, scoped to logged-in user, cleared on logout, and expires after 7 days. _Src: PRD 32.1–32.4; Arch 18; API 9.29._
- **US-081 Enforce tenant and branch isolation** — Every tenant-owned query is scoped by `tenant_id`; Every branch-scoped query is scoped by `tenant_id` and `branch_id`; Requests where authenticated tenant context doesn’t match requested record tenant blocked; Branch-specific data is returned only when user has branch assignment or tenant-wide branch access. _Src: PRD 33.1–33.2; Arch 10; API 5.2–5.3._
- **US-082 Protect sensitive data** — Passwords are hashed with Argon2id or bcrypt cost factor at least 12; Plaintext passwords are never stored, logged, emailed, exported, or exposed; Reset, invite, verification, and refresh tokens are stored as hashes; Access tokens, refresh tokens, reset tokens, and verification tokens are not logged. _Src: PRD 6.5, 33.4–33.5; Arch 22; API 12.2._
- **US-083 Process retry-safe background jobs** — Background jobs support queued, running, succeeded, failed, cancelled, and dead-lettered where implemented; Workers acquire jobs with locks; Workers use correlation IDs; Irreversible operations are idempotent. _Src: PRD 34.7; Arch 19; API 9.28._
- **US-084 Observe API, jobs, integrations, and inventory failures** — API logs include timestamp, level, environment, service name, correlation ID, tenant ID when applicable, user ID when applicable, request path or job type, error code, and sanitized details; Metrics cover API latency/error rate, background job failures/retries, auth failures, authorization denials, inventory transaction failures, notification delivery failures, export duration, and database pool saturation; Alerts exist for critical background failures, tenant deletion failure, payment/receipt transaction failure spike, inventory failure spike, provider failure spike, storage threshold, backup failure, high API error rate, and sustained P95/P99 latency breach; Sensitive request/response bodies are not logged. _Src: PRD 34.6; Arch 23; API 13._
- **US-085 Backup and restore production data** — Database backups are encrypted and run daily; Backup retention is at least 30 days; Restore testing is performed quarterly by platform operators; Backup-retained deleted tenant data is not restored for tenant use except platform-wide disaster recovery. _Src: PRD 33.7–33.8; Arch 25._
- **US-086 Enforce idempotency and concurrency safety** — Required idempotent operations include invoice issuance, payment creation, refund creation, receipt generation, inventory reservation, inventory consumption, adjustment posting, purchase receiving, supplier return posting, supplier payment creation, transfer stock transitions, and tenant deletion execution; Same idempotency key and same request intent returns original result after success; Same key with different request intent returns `idempotency_conflict`; Concurrent document numbering no produce duplicates. _Src: PRD 33.9; Arch 13; API 7, 15.6; Sch 16._
